What‚Äôs all this about Origin and Master?

The ‚Äúorigin‚Äù is the remote location that is housing the code. By default, it is named ‚Äúorigin‚Äù, which is just a convention that arose.

You can set up different remote locations if you like‚ÄîI have set up both GitHub and GitLab on the same repository, I just needed to push my changes to each remote location.

The ‚Äúmaster‚Äù is the default branch that is created when you‚Äôre working on a project. It, too, is a common convention. Branching is a useful feature for developing new or experimental changes in isolation without impacting the existing code.

Command Line Git ‚Äì Key Commands

Creation: git init and git remote

This is how you create a local project!

  (use "git add [file]..." to include in what will be committed)

If you‚Äôre working on a project on a lab computer and a home computer and making changes from both, pull the changes that you pushed from the other computer! Also, if you‚Äôre working with other people you want to make sure you have the latest changes

Best Practices

Using Git correctly helps future you and your teammates

  • Make clean, single-purpose commits
  • Don‚Äôt use --force
  • Write meaningful commit messages
  • Don‚Äôt alter published history
    • I do this A LOT, but only on my personal projects
    • You can re-write git history, but if anyone else is working with your code, it really screws them up and they will have to deal with a lot of errors when trying to merge or pulling
    • Some companies use advanced git configurations to prevent this (look into ‚Äúgit hooks‚Äù)
  • Don‚Äôt commit generated files
    • if your code generates files every time you run it, don‚Äôt bother committing those!
  • Don‚Äôt commit really big files that change often
    • your .git/ folder will be bloated and you‚Äôll have to use a tool like bfg to clean it out
  • Don‚Äôt commit sensitive information
    • This goes without saying, but if you submit passwords or API keys, you‚Äôll need to edit the git history
    • You can‚Äôt just submit a new commit, because someone could look at older commits
    • Use bfg to fix this mistake

Are there other Version Control systems?

Yes, there are actually a few, and the following list is not exhaustive:

  • Git
    • Created in 2005 and currently the most used
    • Decentralized, so you don‚Äôt need a central server
    • Can have local changes and use it completely offline

  • Mercurial
    • Decentralized, like Git
    • Doesn‚Äôt have a Merge feature (some people really don‚Äôt like that)
    • Written in Python
    • Was competing with Git to be used by Linux Kernel, ultimately lost
    • Easier to learn than Git, with better documentation
  • CVS [Concurrent Versions System]
    • a dinosaur from the 80‚Äôs
    • Centralized, runs on your server
    • You ‚Äúcheck out‚Äù code, make changes, then ‚Äúcheck in‚Äù
  • SVN [Apache SubVersioN]
    • A better version of CVS
    • Centralized
    • Not totally uncommon

Naughty Commands

Let‚Äôs talk about what you shouldn‚Äôt do, but might find useful. These all have to do with re-writing history. As a rule of thumb, you know that you‚Äôre doing something bad when you have to use git push --force, so please keep that in mind.

When you use the --force flag, that means your local repo disagrees with the origin. However, if you haven‚Äôt pushed yet, then you can manipulate all your local commits however you want! That‚Äôs the ideal use case for these.

  • git reset --hard [COMMIT HASH]
    • Permanently time travel backwards to a specific commit
    • This will destroy any changes you‚Äôve made since then unless you‚Äôve pushed them
    • Find the commit hash using git log
  • git commit --amend: edit your last commit message
    • I use this a lot for quick adjustments before I push
  • git add [file] && git commit -v --no-edit --amend
    • This adds a file to your last commit and re-submits it
